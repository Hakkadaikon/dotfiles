# 作業の進め方

ponytail(最小・root cause・YAGNI)と micro-commit は別 rule に従う。ここはそれらと重複しない、計画・検証・自律修正・自己改善の指針だけを定める。

## 計画

- 非自明なタスク(3 ステップ以上、または設計判断を含む)は plan mode に入ってから着手する。
- 途中でおかしくなったら、押し進めず止まって計画し直す。
- 計画は検証ステップまで含めて立てる。実装だけでなくどう確かめるかも先に決める。

## subagent の活用

- 調査、探索、並列分析は subagent に委譲し、メインの文脈を汚さない。
- 重い問題には subagent を増やして計算量を投入する。1 subagent につき 1 観点に絞る。
- 委譲先は user グローバルの coder、jp-writer、explorer、code-reviewer、prover(Lean 4 形式検証)、modeler(TLA+ 設計モデル検査)、または内容適応で並列調査する adaptive-explore workflow を使う(リポ側 agents があればそちらを優先)。

## 検証

- 動作を証明できるまでタスクを完了扱いにしない。テスト実行、ログ確認、差分の挙動比較で示す。
- 実装は t-wada スタイルの TDD で駆動する。テストリスト先行 → Red(落ちることを確認)→ Green(最小実装)→ Refactor を小さく回す。設定値・自明な 1 行には持ち込まない(YAGNI)。
- 並行・状態遷移・プロトコルの設計は、コードを書く前に loop-engineering(TLA+)でモデル検査して固める。反例は Gherkin の受け入れ仕様になる。重い検査は modeler に委譲する。状態遷移も並行性も無いなら過剰。
- アルゴリズム・プロトコル・セキュリティ性質で、通常テストより強い保証が要る critical な部分は formal-verification(Lean 4)で証明する。証明済みの性質はそのまま test-first の述語になる。形式化が過剰な所には持ち込まない(YAGNI)。
- 検証の三層を役割で使い分ける: 設計の正しさ → TLA+、実装の数学的証明 → Lean、実装の駆動 → TDD。上 2 層で保証した性質は TDD のテストリスト項目に落とす。無理に結線しない。
- 提出前に自分の成果物を一度疑う。熟練エンジニアがこれを承認するかを自問する。

## 自律修正

- バグ報告を受けたら、手取り足取りを求めず自分で直す。ログ、エラー、失敗テストを手がかりにする。
- CI の失敗は指示を待たずに直す。user の context switch を最小にする。

## 自己改善

- user から訂正を受けたら、そのパターンを `tasks/lessons.md` に書き、同じ失敗を防ぐ自分向けのルールを残す。
- 計画は `tasks/todo.md` にチェック可能な項目で書き、進行に合わせて完了を付け、末尾にレビュー欄を設ける。
- セッション開始時、そのプロジェクトに関係する lessons を見直す。

## 応答の言語

- user への応答は常に日本語で返す(genshijin 規範の範囲内で)。コード、コミットメッセージ、PR は通常記述。
